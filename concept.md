# Fabric Data Agent PoC — 製造業向け製造工程データ対話システム

## 概要

本ドキュメントは、Microsoft Fabric の **Data Agent** 機能を活用し、製造業における製造工程データへの自然言語対話を実現する PoC（概念実証）のコンセプトを定義します。

---

## 1. PoC の目的と背景

### 背景

製造業では、製品の生産にまつわる大量のデータが日々蓄積されています。しかし、そのデータの多くは：

- **構造化データ**（DB テーブル）: 製造指示、工程実績、部材使用実績、品質検査結果
- **非構造化データ**（ファイル）: 製品仕様書、作業手順書、部材カタログ、品質基準書

の 2 種類に分断されており、横断的な情報活用には専門知識や手間がかかっていました。

### 目的

本 PoC では以下を実証します：

1. **自然言語でのデータ対話**: 製造担当者やマネージャーが SQL の知識なしに、製造データへ質問できる
2. **構造化 × 非構造化データの統合**: DB の数値データと仕様書・手順書の文書情報を横断して回答できる
3. **M365 Copilot からのシームレスなアクセス**: 既存の業務ツール（Teams, Outlook など）から自然に利用できる

## 3. 利用するデータ

### 3.1 構造化データ（Fabric Warehouse / SQL）

| テーブル名 | 概要 | 主なカラム |
|---|---|---|
| `dim_products` | 製品マスタ | 製品ID, 製品名, 製品カテゴリ, 規格 |
| `dim_processes` | 製造工程マスタ | 工程ID, 工程名, 工程順序, 担当部門 |
| `dim_parts` | 部材マスタ | 部材ID, 部材名, 部材種別, 仕入先 |
| `dim_workers` | 作業者マスタ | 作業者ID, 氏名, スキルレベル |
| `fact_production_orders` | 製造指示 | 指示ID, 製品ID, 計画数, 計画日 |
| `fact_process_results` | 工程作業実績 | 実績ID, 指示ID, 工程ID, 開始/終了時刻, 作業者ID |
| `fact_parts_usage` | 部材使用実績 | 使用ID, 実績ID, 部材ID, 使用数量 |
| `fact_quality_inspections` | 品質検査結果 | 検査ID, 実績ID, 検査項目, 合否, 測定値 |

### 3.2 非構造化データ（Azure Blob Storage → Fabric OneLake）

| ファイル種別 | 概要 | 格納場所 |
|---|---|---|
| 製品仕様書 | 各製品の電気的仕様・機械的仕様・性能要件 | `product_specs/` |
| 作業手順書 | 各工程の標準作業手順（SOP） | `work_manuals/` |
| 部材カタログ | 各部材の技術的詳細・代替品情報 | `parts_catalog/` |
| 品質基準書 | 各製品・工程の品質許容範囲 | `quality_standards/` |

## 4. 想定ユースケース

### UC-01: 製造実績の確認（生産管理担当）

> 「今週の製品 A の製造完了数と不良率を教えて」

→ `fact_production_orders` × `fact_quality_inspections` を結合して即座に回答

### UC-02: 品質問題の原因調査（品質管理担当）

> 「先月の最終検査で不合格になった製品の工程と、使用した部材を教えて」

→ `fact_quality_inspections` → `fact_process_results` → `fact_parts_usage` を横断して分析

### UC-03: 作業手順の確認（現場作業者）

> 「インバータ制御装置の筐体組立工程の手順を教えて」

→ Azure Blob Storage の作業手順書（SOP）を RAG で検索して回答

### UC-04: 部材情報と実績の統合照会（調達担当）

> 「IGBT モジュールの使用実績と代替品の情報を教えて」

→ `fact_parts_usage` の数値データ + 部材カタログ（非構造化）を統合回答

### UC-05: 製品仕様と検査基準の確認（設計・品質）

> 「サーボモータードライバーの電圧仕様と、それに対応する検査基準値を教えて」

→ 製品仕様書（非構造化）+ `fact_quality_inspections` の実測値を組み合わせ

### UC-06: 管理職向けサマリー生成（製造部長）

> 「今月の生産ラインの稼働状況をサマリーして」

→ 複数テーブルを集計して自然言語でレポート生成

## 6. 期待される効果

### 定量的効果

| 指標 | 現状 | PoC 後の目標 |
|---|---|---|
| 製造実績の確認時間 | 15〜30分（Excel/DB直接操作） | **1〜2分**（自然言語で即座に回答） |
| 品質問題の原因調査時間 | 1〜2時間（複数システム横断） | **10〜20分**（統合クエリで自動分析） |
| 作業手順の確認時間 | 5〜10分（紙/PDF 検索） | **30秒〜1分**（チャットで即座に取得） |
| データ活用ユーザー数 | 限定的（IT・分析担当のみ） | **全製造担当者**（SQL知識不要） |

### 定性的効果

- **データドリブン文化の醸成**: 誰でもデータに基づいた意思決定が可能
- **知識の民主化**: 熟練者の暗黙知（手順書・仕様書）をデジタル化・共有化
- **リアルタイム対応**: 製造ラインでの突発的な問題に即座にデータで対応

## 7. 前提条件・制約

### 前提条件

- Microsoft 365 E3/E5 ライセンス（M365 Copilot 含む）
- Microsoft Fabric ライセンス（F64 以上推奨）
- Azure サブスクリプション（Blob Storage 用）
- Fabric Data Agent 機能が有効なテナント

### 制約・考慮事項

- **データセキュリティ**: 製造データは機密情報を含む可能性があるため、Fabric の行レベルセキュリティ（RLS）を適用
- **データの鮮度**: リアルタイム性が求められる場合は Fabric Direct Lake モードを検討
- **非構造化データの精度**: RAG の精度は文書の品質に依存するため、ドキュメント整備が重要
- **ライセンスコスト**: M365 Copilot は追加ライセンスが必要（$30/ユーザー/月）
